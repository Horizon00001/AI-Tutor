# 相似题来源追踪方案

## 问题分析

当前数据结构中，判断相似题"是谁的相似题"可以通过以下方式：

### 现有字段说明

| 字段 | 含义 |
|------|------|
| `task_id` | 相似题生成任务的唯一ID（UUID） |
| `source_id` | 原始试卷处理任务的ID（DeepSeek任务的task_id） |
| `source_question` | 原题内容（包含title, content, answer, analysis） |

### 现有追踪方式

1. **通过 `source_id`**：相似题任务创建时会存储 `source_id`，指向原始试卷的DeepSeek处理任务task_id
2. **通过 `source_question`**：可以知道具体是哪道题的相似题（通过content比对）
3. **通过文件名中的task_id**：文件命名使用 `{task_id}_similar_questions.json`

### 存在的问题

- `source_question` 只保存了题目内容，**没有保存原始试卷文件名**
- 如果要完整知道"这道相似题来自哪个试卷文件"，需要通过 `source_id` 查询原始试卷处理任务的详情

---

## 解决方案

### 方案：在生成相似题时保存原试卷信息

在 `SimilarQuestionService.generate_similar()` 方法中，保存更多原始试卷信息到 `source_question` 中：

**需要保存的额外信息**：
- `source_file_id`: 原始试卷文件的file_id
- `source_file_name`: 原始试卷文件名
- `question_index`: 该题在原试卷中的索引位置

### 修改文件

1. **services/similar_question_service.py** - 修改 `generate_similar` 方法，在保存时添加原试卷信息

2. **utils/storage.py** - `save_similar_questions` 方法无需修改（它接收完整的source_question字典）

### 具体实现步骤

1. 修改 `api/routes/questions.py` 中创建相似题任务时，传递更多源信息（source_file_id, source_file_name, question_index）

2. 修改 `services/similar_question_service.py` 的 `generate_similar` 方法，将这些信息添加到 `source_question` 中保存

3. 确保保存后的JSON结构包含完整的来源追溯信息

---

## 修改后数据结构示例

```json
{
  "task_id": "相似题生成任务ID",
  "source_question": {
    "title": "1",
    "content": "【单选题】...",
    "answer": "",
    "analysis": "...",
    "source_file_id": "原始试卷file_id",
    "source_file_name": "原试卷文件名.pdf",
    "question_index": 0
  },
  "similar_questions": [...],
  "generated_count": 2,
  "created_at": "1775981792.0"
}
```

这样就可以直接从相似题JSON文件中判断：
1. 这道相似题来自哪个试卷文件（source_file_name）
2. 是试卷中的第几题（question_index）
3. 原始试卷的task_id是什么（需要通过file_id查询）
